# MOCHU-OA 合同标准化整改方案

> 版本 1.0 | 2026-04-07 | 强约束改造 · 不保留自由手写兼容

---

## 一、改造目标

将合同模块从"自由文本录入"改为"模板驱动 + 结构化字段填写"模式。合同类型固定七类，合同正文由管理员上传的模板决定，经办人只能填写模板定义的可编辑字段，固定条款与格式只读不可改，杜绝自定义上传和自由格式正文。

---

## 二、合同类型枚举（固定七类，禁止新增删减）

| 枚举值 | 中文名称 |
|---|---|
| `equipment_purchase` | 设备类采购合同 |
| `material_purchase` | 材料类采购合同 |
| `subcontract` | 专业分包采购合同 |
| `labor` | 劳务施工合同 |
| `tech_service` | 技术服务合同 |
| `software` | 软件工程合同 |
| `other` | 其他类合同 |

**约束规则**：
- 后端 `ContractTypeEnum` 硬编码，不读数据库，不提供增删接口。
- 前端下拉项从 `GET /api/v1/enums/contract-types` 获取，禁止手写输入。
- `biz_project.contract_type` 与 `biz_contract.contract_type` 统一改用上述枚举，`VARCHAR(30)` 足够。

---

## 三、项目立项及业务字段枚举化

凡现有自由文本字段属于固定枚举范围的，统一改为下拉选择：

| 模块 | 字段 | 改为枚举/下拉 | 值域 |
|---|---|---|---|
| 项目 | `project_type` | 已是下拉 | 1虚拟 / 2实体 |
| 项目 | `contract_type` | 改为下拉 | 七类枚举（同上） |
| 合同 | `contract_type` | 改为下拉 | 七类枚举（同上），废弃 income/expense |
| 合同 | `status` | 已是下拉 | draft/pending/approved/rejected/terminated |
| 采购 | `purchase_type` | 改为下拉 | 根据业务确定固定选项 |

前端所有枚举字段使用 `<el-select>` 且 `:allow-create="false"`。

## 四、合同模板体系设计

### 4.1 模板管理功能（管理员）

管理员可对每种合同类型维护一套模板，支持：

| 功能 | 说明 |
|---|---|
| 上传 | 上传 `.docx` 模板文件至 MinIO，解析占位符提取可编辑字段定义 |
| 版本 | 每次上传产生新版本号（自增），保留历史版本 |
| 启停 | 每类型同一时刻只允许一个启用版本 |
| 生效时间 | `effective_from` / `effective_until`，可设定未来生效 |
| 预览 | 将模板转为 HTML/PDF 供在线预览 |
| 下载 | 下载原始 `.docx` 模板文件 |
| 替换 | 上传新版本并停用旧版本（原子操作） |
| 审计日志 | 记录上传/启用/停用/替换/删除操作，含操作人、时间、版本号 |

### 4.2 模板与合同类型绑定关系

```
合同类型（7类） ──1:N──▶ 模板主记录 ──1:N──▶ 模板版本
                                    └──1:N──▶ 模板字段定义
```

- 一个合同类型对应一条模板主记录。
- 模板主记录下挂多个版本（历史追溯）。
- 模板字段定义挂在版本下（不同版本的字段可能不同）。

### 4.3 模板字段定义（结构化填写区）

模板中的可编辑字段通过占位符标记（如 `{{partyA}}`），上传后后端解析提取。每个字段定义包含：

| 属性 | 说明 |
|---|---|
| `field_key` | 字段标识，如 `party_a`、`amount`、`delivery_date` |
| `field_name` | 中文名称，如"甲方名称" |
| `field_type` | 类型：`text` / `number` / `date` / `select` / `textarea` |
| `required` | 是否必填 |
| `options_json` | select 类型的选项列表（JSON 数组） |
| `default_value` | 默认值 |
| `sort_order` | 字段排序 |
| `placeholder` | 占位提示文字 |
| `max_length` | 文本最大长度 |
| `validation_rule` | 正则校验规则（可选） |

**填写区控制原则**：
- 下划线仅作展示效果（模板 HTML 渲染中对空值字段显示下划线占位）。
- 用户只能填写 `sys_contract_tpl_field` 中已定义的字段。
- 前端根据字段定义动态渲染表单控件，不渲染自由文本编辑器。
- 静态正文（模板固定条款）只读展示，不可编辑。
- 后端校验提交数据的 key 集合必须是字段定义的子集，拒绝未定义字段。

## 五、数据模型

### 5.1 新增表

```sql
-- 合同模板主表
CREATE TABLE sys_contract_tpl (
  id              INT           PRIMARY KEY AUTO_INCREMENT,
  contract_type   VARCHAR(30)   NOT NULL COMMENT '绑定的合同类型枚举',
  tpl_name        VARCHAR(200)  NOT NULL COMMENT '模板名称',
  description     VARCHAR(500)  DEFAULT NULL COMMENT '模板说明',
  status          TINYINT       NOT NULL DEFAULT 1 COMMENT '1启用/0停用',
  creator_id      INT           NOT NULL,
  created_at      DATETIME      NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at      DATETIME      DEFAULT NULL ON UPDATE CURRENT_TIMESTAMP,
  deleted         TINYINT       NOT NULL DEFAULT 0,
  UNIQUE KEY uk_tpl_type (contract_type, deleted),
  KEY idx_tpl_status (status)
) COMMENT '合同模板主表';

-- 模板版本表
CREATE TABLE sys_contract_tpl_version (
  id              INT           PRIMARY KEY AUTO_INCREMENT,
  tpl_id          INT           NOT NULL COMMENT '模板主表ID',
  version_no      INT           NOT NULL DEFAULT 1 COMMENT '版本号(递增)',
  file_path       VARCHAR(500)  NOT NULL COMMENT 'MinIO存储路径',
  file_name       VARCHAR(200)  NOT NULL COMMENT '原始文件名',
  file_md5        VARCHAR(64)   NOT NULL COMMENT '文件MD5(防篡改)',
  html_cache      LONGTEXT      DEFAULT NULL COMMENT '解析后的HTML缓存(含占位符)',
  status          TINYINT       NOT NULL DEFAULT 1 COMMENT '1启用/0停用',
  effective_from  DATETIME      DEFAULT NULL COMMENT '生效开始时间',
  effective_until DATETIME      DEFAULT NULL COMMENT '生效截止时间',
  creator_id      INT           NOT NULL,
  created_at      DATETIME      NOT NULL DEFAULT CURRENT_TIMESTAMP,
  KEY idx_tpl_version (tpl_id, version_no),
  KEY idx_tpl_version_status (tpl_id, status)
) COMMENT '合同模板版本表';

-- 模板字段定义表
CREATE TABLE sys_contract_tpl_field (
  id              INT           PRIMARY KEY AUTO_INCREMENT,
  version_id      INT           NOT NULL COMMENT '模板版本ID',
  field_key       VARCHAR(50)   NOT NULL COMMENT '字段标识(占位符名)',
  field_name      VARCHAR(100)  NOT NULL COMMENT '字段中文名',
  field_type      VARCHAR(20)   NOT NULL DEFAULT 'text' COMMENT 'text/number/date/select/textarea',
  required        TINYINT       NOT NULL DEFAULT 0 COMMENT '是否必填',
  options_json    VARCHAR(1000) DEFAULT NULL COMMENT 'select选项JSON',
  default_value   VARCHAR(500)  DEFAULT NULL,
  sort_order      INT           NOT NULL DEFAULT 0,
  placeholder     VARCHAR(200)  DEFAULT NULL,
  max_length      INT           DEFAULT NULL,
  validation_rule VARCHAR(200)  DEFAULT NULL COMMENT '正则校验规则',
  KEY idx_field_version (version_id, sort_order)
) COMMENT '合同模板字段定义表';

-- 合同字段值表（实例级）
CREATE TABLE biz_contract_field_value (
  id              INT           PRIMARY KEY AUTO_INCREMENT,
  contract_id     INT           NOT NULL COMMENT '合同ID',
  field_key       VARCHAR(50)   NOT NULL COMMENT '字段标识',
  field_value     TEXT          DEFAULT NULL COMMENT '填写值',
  created_at      DATETIME      NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at      DATETIME      DEFAULT NULL ON UPDATE CURRENT_TIMESTAMP,
  UNIQUE KEY uk_contract_field (contract_id, field_key),
  KEY idx_field_contract (contract_id)
) COMMENT '合同字段值表';

-- 模板操作审计日志
CREATE TABLE sys_contract_tpl_audit (
  id              INT           PRIMARY KEY AUTO_INCREMENT,
  tpl_id          INT           NOT NULL COMMENT '模板ID',
  version_id      INT           DEFAULT NULL COMMENT '版本ID',
  action          VARCHAR(30)   NOT NULL COMMENT 'upload/enable/disable/replace/delete',
  detail          VARCHAR(500)  DEFAULT NULL COMMENT '操作详情',
  operator_id     INT           NOT NULL,
  operated_at     DATETIME      NOT NULL DEFAULT CURRENT_TIMESTAMP,
  KEY idx_audit_tpl (tpl_id),
  KEY idx_audit_time (operated_at)
) COMMENT '合同模板审计日志表';
```

### 5.2 修改现有表

```sql
-- biz_contract.contract_type 值域从 income/expense 改为七类枚举

```

                                     │

## 七、前端改造要点

### 7.1 枚举下拉配置

- 合同类型：调用枚举接口渲染 `<el-select>`，禁止 `allow-create`。
- 项目立项表单中的"合同类型"同步改为该下拉。
- 所有状态字段保持现有 `<el-select>` 模式。

### 7.2 模板管理页面（新增）

路径：`/system/contract-tpl`，添加到系统管理子菜单。

功能：
- 模板列表表格：类型、名称、当前版本、状态、操作（上传新版本/启停/预览/审计日志）。
- 上传对话框：文件选择（仅 `.docx`）+ 上传后显示解析出的字段列表，可调整字段属性。
- 预览对话框：iframe 或 HTML 渲染模板内容。
- 审计日志对话框：按模板 ID 查询日志列表。

### 7.3 合同签订页面改造

**创建流程**：
1. 选择合同类型（下拉）→ 自动加载该类型当前启用模板。
2. 左侧显示模板正文预览（只读 HTML，可编辑字段用下划线占位高亮显示）。
3. 右侧/下方显示结构化表单（根据模板字段定义动态渲染）。
4. 提交时前端校验必填项 → 调用 `POST /api/v1/contracts` → 提示"已提交审批"。

**填写区控制**：
- 根据 `field_type` 渲染对应组件：`text`→`el-input`、`number`→`el-input-number`、`date`→`el-date-picker`、`select`→`el-select`、`textarea`→`el-input[type=textarea]`。
- 下划线效果：模板 HTML 中 `{{field_key}}` 占位符渲染为 `<span class="tpl-field-slot">` + 下划线样式，填入值后替换显示。
- 禁止 `contenteditable`、富文本编辑器、手动输入框介入模板正文区域。

### 7.4 校验与提示

| 场景 | 前端行为 |
|---|---|
| 该类型无启用模板 | 禁用创建按钮，提示"该合同类型尚未配置模板，请联系管理员" |
| 必填字段未填 | 表单校验拦截，红色提示 |
| 字段超长/格式不符 | 实时校验提示 |
| 模板已停用 | 创建时后端返回错误，前端提示"模板已停用" |

## 八、后端核心逻辑

### 8.1 模板上传与解析

```
上传 .docx → MinIO 存储 → 计算 MD5 → poi-tl 解析占位符 {{...}}
→ 提取字段列表 → 写入 sys_contract_tpl_field → 生成 HTML 缓存
→ 记录审计日志
```

技术选型：Apache POI + poi-tl（模板引擎），用于解析 `.docx` 中的 `{{placeholder}}` 占位符。

### 8.2 合同创建校验链

```
1. 校验 contract_type 属于七类枚举
2. 查找该类型当前启用且在生效时间内的模板版本
3. 若无启用模板 → 拒绝创建
4. 加载模板字段定义列表
5. 校验提交字段：
   a. 所有 key 必须在字段定义中存在（拒绝未定义字段 → 防篡改）
   b. required=1 的字段必须有值
   c. 按 field_type 校验格式（number/date/select 选项范围）
   d. max_length 校验
   e. validation_rule 正则校验
6. 写入 biz_contract（status=pending, template_id, tpl_version_id）
7. 批量写入 biz_contract_field_value
8. 调用 ApprovalService.submitForApproval("contract", id, userId, bizContext)
9. 记录审计日志
```

### 8.3 防篡改机制

- 模板文件存 MD5，下载/导出时校验 MD5 一致。
- 合同字段值表 `biz_contract_field_value` 仅允许 `draft`/`rejected` 状态修改。
- 合同实例冻结 `tpl_version_id`，审批中/通过后不允许更换模板版本。
- 字段值提交时校验 key 白名单（模板字段定义），拒绝注入未定义字段。

### 8.4 幂等控制

- 合同创建：`contract_no` 唯一键 + 同一 `biz_type`+`biz_id` 不允许重复提交审批（`ApprovalService` 已有检查）。
- 模板上传：同一 MD5 文件不重复上传至 MinIO（检查 `file_md5` 列）。

### 8.5 导出

合同导出 = 模板文件 + 字段值 → poi-tl 填充生成 `.docx` → 可选转 PDF。

## 九、场景覆盖矩阵

| 场景 | 处理方式 |
|---|---|
| 管理员配置模板 | 模板管理页上传 `.docx`，系统解析字段，管理员确认启用 |
| 立项选择合同类型 | 项目表单"合同类型"改为七类枚举下拉 |
| 经办人按模板填写 | 创建合同时自动加载模板，动态渲染字段表单 |
| 审批查看 | 审批人点击详情查看模板 + 填充值的预览 HTML |
| 模板更新 | 上传新版本，新合同用新版本，已有合同冻结旧版本 |
| 历史兼容 | 已有合同的 `tpl_version_id` 不变，仍可预览/导出 |
| 未配置模板 | 前端禁用创建按钮，后端拒绝并返回明确错误 |
| 模板停用 | 已停用模板不可用于新建合同，已签合同不受影响 |
| 字段缺失 | 后端校验 required 字段，返回缺失字段列表 |
| 越权修改 | `pending`/`approved` 状态禁止修改字段值，后端拦截 |
| 版本失效 | 查询启用版本时校验 `effective_from`/`effective_until` 时间窗 |
| 未定义字段提交 | 后端白名单校验，拒绝未定义 key |

## 十、历史数据迁移

### 10.1 迁移策略

```
现有 biz_contract 数据
  ├─ contract_type = income/expense
  │     └─ 需映射到七类枚举
  ├─ 无 template_id / tpl_version_id
  │     └─ 保留 NULL，标记为"历史自由文本合同"
  └─ 自由文本字段（party_a/party_b/remark 等）
        └─ 尝试映射到模板字段 → 写入 biz_contract_field_value
```

### 10.2 映射规则

| 旧值 | 映射逻辑 |
|---|---|
| `income` | 根据 `remark`/`contract_name` 关键词匹配七类之一，无法匹配则归入 `other` |
| `expense` | 同上 |

### 10.3 人工复核池

无法自动映射的合同进入复核池：
- 后端提供 `GET /api/v1/contracts/migration/pending` 查询待复核列表。
- 管理员逐条确认合同类型 → `PATCH /api/v1/contracts/{id}/migrate` 指定新类型。
- 全部复核完成后，`contract_type` 旧枚举值不再合法，后端校验拒绝。

| 模板替换→新合同用新版本→旧合同不受影响 | 版本隔离 |
| 无模板时创建合同 | 拒绝并提示 |
| 越权修改字段值 | 拒绝并返回 403 |
| 并发创建同编号合同 | 幂等唯一键拦截 |

### 11.3 验收标准

- [ ] 七类合同类型下拉全覆盖，无自由输入入口
- [ ] 管理员可完成模板全生命周期管理（上传/版本/启停/预览/下载/审计）
- [ ] 经办人创建合同时自动套用模板，只能填写已定义字段
- [ ] 模板正文区域完全只读，不可编辑
- [ ] 审批人可预览完整合同（模板 + 填充值）
- [ ] 历史合同可正常查看，迁移数据完整
- [ ] 未配置模板/模板停用/字段缺失/越权修改等异常场景全部拦截

## 十二、实施原则

1. **强约束，不兼容**：本次改造为破坏性变更，不保留自由手写入口。合同创建必须走模板通道。
2. **历史清洗，人工兜底**：旧 `income`/`expense` 数据必须迁移映射，无法映射的进入人工复核池，不允许跳过。
3. **模板驱动，字段白名单**：前端渲染由模板字段定义驱动，后端校验由字段白名单控制，双重保障。
4. **版本冻结，互不影响**：合同实例绑定签订时的模板版本，后续模板更新不影响已有合同。
5. **接口对齐，颗粒度一致**：所有列表 `PageResult`，状态变更 `PATCH`，权限 `@PreAuthorize`，DTO 校验 `@Valid`，响应 `R<T>` 统一包装。
6. **先建模板，后开合同**：上线后管理员必须先为至少一个合同类型配置并启用模板，才能创建合同。
7. **审计可追溯**：模板操作全留痕，合同字段值变更可通过 `updated_at` + 审批记录追溯。
