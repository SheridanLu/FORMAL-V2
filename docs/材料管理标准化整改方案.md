# MOCHU-OA 材料管理标准化整改方案

> 版本：V1.0 | 日期：2026-04-08 | 状态：待实施

---

## 一、功能需求清单

### 1.1 枚举约束

| 枚举项 | 固定值 | 约束 |
|--------|--------|------|
| 材料分类 | `设备`、`材料`、`人工` | 三类固定，禁止新增/删减/修改/自定义 |
| 税率(%) | `0`、`1`、`3`、`6`、`9`、`13` | 六档固定，下拉选择 |
| 计量单位 | `米`、`吨`、`卷`、`箱`、`台`、`项`、`只`、`套` | 八项固定，下拉选择 |
| 基准价 | 统一为**含税价**，字段标识 `base_price_with_tax` | 明确语义，禁止歧义 |

### 1.2 核心需求

| 编号 | 需求 | 说明 |
|------|------|------|
| M-01 | 分类下拉强约束 | 前后端同时收口，只允许三类枚举值 |
| M-02 | 税率下拉强约束 | 六档固定税率，下拉选择 |
| M-03 | 单位下拉强约束 | 八项固定单位，下拉选择 |
| M-04 | 基准价字段重命名 | `base_price` → `base_price_with_tax`，明确含税语义 |
| M-05 | 批量在线表格录入 | 替代单条弹窗录入，支持行内编辑 |
| M-06 | 动态增删行 | 底部"添加行"按钮 + 每行删除按钮 |
| M-07 | 一次提交多条 | 批量保存接口，事务控制 |
| M-08 | 每行枚举约束 | 每行分类/税率/单位均受枚举限制 |
| M-09 | 编码自动生成 | 后端按规则生成 `material_code`，前端不可编辑 |
| M-10 | 去重校验 | 同名+同规格+同单位视为重复，前后端双重校验 |

### 2.2 下拉配置

| 字段 | 组件 | 数据来源 | 配置 |
|------|------|----------|------|
| 分类 | `el-select` | 前端硬编码 `['设备','材料','人工']` | `filterable=false`，禁止输入过滤 |
| 单位 | `el-select` | 前端硬编码 `['米','吨','卷','箱','台','项','只','套']` | 同上 |
| 税率 | `el-select` | 前端硬编码 `[0,1,3,6,9,13]` | label 显示 `0%`~`13%`，value 为数字 |

| 状态 | `el-select` | `['启用','停用']` | 新增时默认"启用"，不在批量表格中显示 |

### 2.3 必填校验

| 字段 | 规则 | 触发时机 | 错误提示 |
|------|------|----------|----------|
| 材料名称 | 非空，长度 2~100 | blur + 提交 | "请输入材料名称(2~100字)" |
| 分类 | 必须为三类之一 | change + 提交 | "请选择材料分类" |
| 单位 | 必须为八项之一 | change + 提交 | "请选择计量单位" |
| 含税基准价 | 非空，≥0，最多2位小数 | blur + 提交 | "请输入含税基准价" |
| 税率 | 必须为六档之一 | change + 提交 | "请选择税率" |

### 2.4 格式校验

| 校验项 | 规则 | 处理 |
|--------|------|------|
| 基准价格式 | 数字，≥0，精度2位 | `el-input-number :precision="2" :min="0"` |
| 名称长度 | 2~100字符 | `:maxlength="100"` + 前端校验 |
| 规格型号长度 | 0~200字符 | `:maxlength="200"` |

### 2.5 重复校验

- **前端校验**：提交前遍历所有行，检测 `名称+规格型号+单位` 三元组是否重复，重复行标红高亮
- **后端校验**：批量创建时查询数据库，返回已存在条目的行号和具体字段

### 2.6 错误高亮

- 校验不通过的单元格边框标红（`class="is-error"`）
- 在行尾显示错误摘要（tooltip 或 inline 文字）
- 后端返回的错误按行号定位，逐行标注

### 2.7 批量保存

- 按钮文案：`批量保存(N条)`，N 为当前有效行数
- 提交前执行全量前端校验，任何一行不通过则阻止提交并滚动到第一个错误行
- 提交后根据后端响应：
  - 全部成功：关闭弹窗，刷新列表，显示 `成功新增N条材料`
  - 部分失败：保留弹窗，成功行移除，失败行标红并显示后端错误原因
  - 全部失败：保留弹窗，所有行标红

### 2.8 取消编辑

- 点击"取消"或关闭弹窗：若有已填写内容，弹出确认框 `已填写的数据将丢失，确定取消？`
- 确认后清空表格数据，关闭弹窗

### 2.9 成功/失败提示

| 场景 | 提示 | 类型 |
|------|------|------|
| 全部成功 | `成功新增N条材料` | ElMessage.success |
| 部分失败 | `成功N条，失败M条，请检查标红行` | ElMessage.warning |
| 全部失败 | `提交失败，请检查标红行` | ElMessage.error |
| 删除成功 | `删除成功` | ElMessage.success |
| 网络异常 | `网络异常，请重试` | ElMessage.error |

### 2.10 空数据处理

- 批量弹窗打开时默认 1 行空行，分类/单位/税率有默认值（分类=材料，单位=项，税率=13）
- 提交时过滤全空行（所有字段均为空/默认值的行不提交）
- 至少保留 1 行，不允许删除到 0 行

### 2.11 权限控制

| 操作 | 权限码 | 说明 |
|------|--------|------|
| 查看列表 | `material:view` | 搜索、分页、导出 |
| 新增材料 | `material:edit` | "新增材料"按钮可见性 |
| 编辑材料 | `material:edit` | 编辑按钮可见性 |
| 删除材料 | `material:edit` | 删除按钮可见性 |

- 前端通过 `v-permission` 指令控制按钮显隐
- 后端通过 `@PreAuthorize` 注解拦截

### 3.4 入参校验

```java
@Data
public class MaterialBatchDTO {
    @NotNull @Size(min = 1, max = 100, message = "单次批量创建1~100条")
    private List<MaterialBatchItem> items;

    @Data
    public static class MaterialBatchItem {
        @NotBlank(message = "材料名称不能为空")
        @Size(min = 2, max = 100)
        private String materialName;

        @Size(max = 200)
        private String specModel;

        @NotBlank(message = "分类不能为空")
        private String category;      // 枚举校验在Service层

        @NotBlank(message = "单位不能为空")
        private String unit;           // 枚举校验在Service层

        @NotNull(message = "含税基准价不能为空")
        @DecimalMin("0")
        private BigDecimal basePriceWithTax;

        @NotNull(message = "税率不能为空")
        private Integer taxRate;       // 枚举校验在Service层
    }
}
```

### 3.5 Service 层处理流程

```
MaterialService.batchCreate(dto, operatorId):
  1. 遍历 items，逐行校验:
     a. MaterialCategoryEnum.isValid(category)  → 否则记录错误
     b. MaterialUnitEnum.isValid(unit)          → 否则记录错误
     c. MaterialTaxRateEnum.isValid(taxRate)    → 否则记录错误
     d. materialName 长度 2~100                 → 否则记录错误
  2. 批量查询数据库去重:
     SELECT material_name, spec_model, unit FROM biz_material_base
     WHERE (material_name, spec_model, unit) IN ((...), (...))
     → 已存在的记录错误
  3. 行内去重: items 内部 (name+spec+unit) 三元组不可重复
  4. 收集所有错误 → 若全部失败直接返回
  5. @Transactional: 仅对通过校验的行执行批量 INSERT
     - 为每行生成 material_code (NoGeneratorService)
     - 设置 status = "active", creator_id = operatorId
  6. 返回 BatchResult(successCount, failCount, errors)
```

### 3.6 事务控制

- 成功行在**同一事务**内批量插入，保证原子性
- 失败行不影响成功行（先分离校验，再统一插入通过的行）
- 若插入过程中发生异常，整批回滚，返回系统错误

### 3.7 幂等控制

- 请求头 `X-Idempotency-Key` 由前端 Axios 拦截器自动生成
- 后端通过 Redis 或数据库记录幂等键，5 分钟内重复请求返回首次结果
- 防止网络重试导致重复创建

### 3.8 错误明细返回

```java
@Data
public class BatchResult {
    private int successCount;
    private int failCount;
    private List<BatchError> errors;

    @Data
    @AllArgsConstructor
    public static class BatchError {
        private int row;        // 行号(从1开始)
        private String field;   // 错误字段
        private String message; // 错误原因
    }
}
```

### 3.9 审计字段

- `creator_id`：由 `SecurityUtils.getCurrentUserId()` 填充
- `created_at`：MyBatis-Plus `FieldFill.INSERT` 自动填充
- `updated_at`：MyBatis-Plus `FieldFill.UPDATE` 自动填充
- 逻辑删除：`@TableLogic private Integer deleted`

### 3.10 异常回滚

| 异常类型 | 处理 |
|----------|------|
| 枚举校验失败 | 记入 errors，不中断 |
| 去重校验失败 | 记入 errors，不中断 |
| DTO 格式校验失败 | GlobalExceptionHandler → 400 |
| SQL 异常 | 整批回滚 → 500 |
| 权限不足 | AccessDeniedException → 403 |

### 3.11 单条编辑接口调整

```
PUT /api/v1/materials/{id}
```

入参同样受枚举约束，Service 层校验 category/unit/taxRate 必须为合法枚举值。

## 六、实施原则

### 6.1 强约束声明

本次改造为**强约束标准化改造**，实施以下不可协商原则：

1. **枚举硬编码**：分类、税率、单位三项枚举在前后端代码中硬编码为 Java Enum 和 JS 常量，不通过数据库配置、不提供管理界面、不允许运行时修改
2. **禁止自由手写**：所有枚举字段必须使用下拉选择组件（`el-select`），前端禁止配置 `filterable`/`allow-create` 等属性
3. **前后端同时收口**：前端下拉约束 + 后端枚举校验双重保障，任何一方不可单独放开
4. **不保留兼容入口**：原有自由文本输入的 `category`/`unit` 字段彻底改为枚举约束，不提供"其他"选项
5. **历史数据清洗**：已有不符合枚举规范的数据须通过 SQL 脚本映射清洗，无法映射的进入人工复核

### 6.2 接口颗粒度对齐

- 前端使用的下拉选项值必须与后端枚举值**完全一致**（字符串精确匹配）
- 前端 Axios 拦截器的 camelCase → snake_case 转换适用于顶层字段名，枚举值本身（中文字符）不受影响
- 后端 Jackson SNAKE_CASE 策略仅影响字段名，枚举值按原值序列化
- 批量接口的错误明细使用 `row`（行号）定位，前端按行号与表格行对应

### 6.3 废弃清理

| 废弃项 | 替代 |
|--------|------|
| `POST /api/v1/materials` 单条创建 | `POST /api/v1/materials/batch` |
| 前端单条新增弹窗 | 批量编辑表格弹窗 |
| `base_price` 字段 | `base_price_with_tax` |
| 自由文本 `category` | 枚举下拉 |
| 自由文本 `unit` | 枚举下拉 |

### 6.4 数据库迁移顺序

```
1. ALTER TABLE 修改字段 (base_price → base_price_with_tax, 加 tax_rate)
2. UPDATE 历史数据清洗 (category/unit 映射)
3. ADD UNIQUE INDEX 去重索引
4. 验证数据完整性
```

### 6.5 Entity 对齐

- `BizMaterialBase.java` 中 `basePrice` → `basePriceWithTax`
- 新增 `taxRate` 字段（Integer）
- 确保 `@TableName("biz_material_base")` 与表名一致
